Projets 2ème Année

Projet RFTG



Contexte de réalisation du projet RFTG

Dans le cadre de la deuxième année de BTS SIO (option SLAM), nous avons développé en équipe une application web de gestion de stock de DVD destinée à un village de vacances. Baptisée Mario, elle est dédiée aux gérants et membres habilités (staff) pour assurer le suivi des films disponibles et l’inventaire des exemplaires. Connectée à la base de données partagée “Peach”, elle dialogue également avec l’application mobile Luigi (côté clients).

Dans ce projet nous avons analysé les besoins, conçu les maquettes, le MCD (modèle conceptuel de données, mise en œuvre concrète de l'application web (Gestion/Administrateur) et de l'application mobile (Location des films/ Clients). J'ai effectué les test de vérification et gestion des erreurs (débogage, journalisation). Le déploiement se fera via un serveur fournie par notre hiérarchie.

Projet RFTG : Application Web - Gestion des stocks (MARIO)

  • Contexte de réalisation

    Dans le cadre de la deuxième année de BTS SIO (option SLAM), nous avons développé en équipe une application web de gestion de stock de DVD destinée à un village de vacances. Baptisée Mario, elle est dédiée aux gérants et membres habilités (staff) pour assurer le suivi des films disponibles et l’inventaire des exemplaires. Connectée à la base de données partagée “Peach”, elle dialogue également avec l’application mobile Luigi (clients).

    Ce projet répond aux attendus du référentiel BTS SIO : analyse des besoins, conception (maquettes, MCD), mise en œuvre concrète, test de vérification et gestion des erreurs (débogage, journalisation).
  • Objectifs

    • Permettre la connexion des gérants avec leurs identifiants (table staff).
    • Gérer la liste des films (consultation, recherche, modification, suppression).
    • Administrer l’inventaire des exemplaires (visualisation, ajout, suppression).
    • Assurer la communication avec le back-end Spring Boot exposant des API REST.
    • Garantir l’accès à la base “Peach” via des services web.
  • Rôles dans le projet

    • Conception et réalisation du front-end Laravel (Blade, CSS/Tailwind, Guzzle).
    • Tests fonctionnels.
    • Collaboration à la modélisation de la base (MCD Looping).
    • Participation à la création des web services en Spring Boot (Java).
    • Analyse des logs Laravel et Apache pour le débogage.
  • Conception technique

    Maquettes

    Réalisées avec Miro, elles définissent les écrans principaux (connexion, liste des films, formulaires de création, inventaire…).

    MCD et base de données



    Le Modèle Conceptuel de Données (Looping) inclut les entités film, inventory, store et staff. La base “Peach” (MySQL) centralise les données pour Mario et Luigi.

    Back-end Spring Boot



    Des API REST gèrent la récupération et la mise à jour des données (ex. /toad/film/all, /toad/staff/getByEmail). Les logs Java permettent de tracer les requêtes et les erreurs.

    Front-end Laravel

    Utilisé pour créer les vues (Blade), envoyer les requêtes HTTP via Guzzle, gérer la session utilisateur (login/logout) et appliquer un style avec TailwindCSS. Lorsque la structure de la table staff ne correspond pas aux conventions Eloquent, des requêtes manuelles sont utilisées.

  • Fonctionnalités principales

    Page de connexion

    Authentification par la table staff (mots de passe non hachés pour simplifier les tests locaux). Eloquent non utilisable à cause d’un schéma de table atypique.



    Liste des films

    Recherche dynamique (GET). Cartes de films (titre, description, année). Actions modifier/supprimer avec confirmation.



    Inventaire

    Tableau listant les exemplaires disponibles. Formulaire d’ajout d’exemplaire (envoi à l’API) avec gestion d’erreurs (logs Laravel).



  • Tests et débogage

    • Usage de dd(), Log::info() et try/catch dans Laravel
    • Suivi des appels Guzzle sur console Apache
    • REST avec Postman
    • Scénarios utilisateurs validés manuellement (TP de vérification)
  • Difficultés rencontrées

    • Structure staff incompatible avec Eloquent (pas d’ID auto, ni timestamps)
    • Conflits de ports sur XAMPP, blocages aléatoires
    • Environnements hétérogènes (droits restreints, installations parfois instables)
  • Conclusion

    Le projet Mario illustre la mise en œuvre des compétences du BTS SIO : travail en équipe, conception MCD, création d’interfaces (HTML/CSS/Tailwind), intégration d’API REST, tests et adaptation aux contraintes réelles. L’application offre un outil fiable pour gérer un stock de DVD dans un contexte pédagogique, tout en respectant les objectifs de l’option SLAM.

Projet RFTG : Application Mobile - Location des DVDs (Luigi)

  • Contexte de réalisation

    Dans la continuité du projet RFTG, l’application Luigi représente la version mobile client du système de gestion de DVD développé en 2e année de BTS SIO option SLAM. Tandis que Mario était destiné aux gérants (back-office), Luigi permet aux clients d’un village de vacances de consulter les films disponibles, de les ajouter à un panier virtuel et de finaliser leur location.

    Le projet se connecte à la base de données partagée “Peach” via les webservices REST du back-end Spring Boot — le même socle que pour l’application web Mario.
  • Objectifs

    • Offrir une interface simple et responsive pour consulter les films.
    • Permettre l’ajout de films à un panier (via filmId).
    • Valider la location d’un ou plusieurs films.
    • Envoyer les informations vers inventory et rental via les webservices existants.
    • Adapter dynamiquement l’URL de l’API (pas de hard-coding).
    • Respecter la structure imposée du backend, sans modifications.
  • Outils et technologies utilisés

    • Android Studio Giraffe (2022.3.1 Patch 1) : IDE principal
    • Java (Android SDK 33) : langage de développement mobile
    • Spring Boot (REST API partagée, tronc commun RFTG)
    • MySQL (base Peach)
    • HTTPURLConnection / Threads : appels asynchrones
    • Figma / Miro : maquettes d’écrans
    • ArrayList<Integer> : stockage temporaire du panier
    • JSON / JSONObject : traitement des réponses API
    • Android Toasts / Logcat : debug et feedback
  • Fonctionnement technique

    Architecture

    L’application mobile se compose d’une activité principale pour la liste des films et la navigation, d’un panier gérant les ajouts, et de plusieurs activités dédiées (connexion, configuration d’URL, détails d’un film, validation de la location). Le back-end Spring Boot expose des endpoints /toad/film, /toad/inventory, /toad/rental pour lire et modifier les données (table inventory, rental).

    Exemple de flux

    L’utilisateur ajoute un film (filmId) au panier. Au moment de valider la location :

    • L’app contacte /inventory/available/getById pour trouver un inventory_id.
    • Elle appelle ensuite /rental/add pour insérer la location dans la base.
    • Les données sont insérées réellement, sans simulation.
  • Particularités techniques

    Asynchrone (Thread Java)

    Les appels aux APIs se font dans des threads séparés (via Thread ou AsyncTask), pour éviter de bloquer l’UI Thread. Indispensable pour une application fluide et responsive.

    URL dynamique

    Chaque écran récupère une URL configurée (selectedURL) lors de l’ouverture (via Intent), afin d’éviter les URLs en dur. Cela facilite le test sur différents réseaux (IP locale, distant).

  • Difficultés rencontrées

    • Gestion du stock réel (inventory) et règles backend imposées par Spring Boot (pas de modif du code serveur).
    • Structure imposée : impossible de contourner l’API, toutes les opérations passent par les webservices.
    • Nécessité d’associer filmId (côté client) au title pour affichage, sans perdre la correspondance en base pour rental.
    • Validation multiple : enchaînement de plusieurs appels asynchrones.
  • Structuration du projet Android

    • MainActivity : point d’entrée de l’app (liste des films, navigation)
    • LoginActivity : écran de connexion (table staff), ou configuration d’URL + IP
    • URLConfigActivity : choix/édition de l’URL dynamique (env de dev ou prod)
    • FilmDetailsActivity : détails d’un film (via filmId) + bouton “Ajouter au panier”
    • PanierActivity : affichage des films ajoutés (avec ArrayList<Integer>), validation
    • Panier.java : singleton statique pour stocker les filmId
    • Layouts XML + ListView / ArrayAdapter / Toast / Log / Intent
  • Résultat obtenu

    L’utilisateur peut consulter les films, les ajouter au panier et finaliser la location. Les enregistrements sont insérés en base de manière réelle (tables inventory, rental), et l’interface reste fluide grâce aux appels asynchrones. Le tout se connecte au tronc commun (Spring Boot), garantissant une logique centralisée et évitant la duplication.

    • Projet (2ème année) : La Boîte à idées

    • Contexte et objectifs

      Dans le cadre de mon BTS SIO, j’ai développé une application web nommée “La Boîte à idées”, en utilisant Laravel, Composer, Node.js et XAMPP. L’objectif est de proposer une plateforme où les utilisateurs peuvent poster leurs idées, commenter celles des autres et se conformer à des règles métier strictes. En parallèle, j’ai exploré des bases de la sécurisation des applications web et j’ai intégré un processus de nettoyage/filtrage de données compromises (application concurrente fictive).
    • Règles métier

      • Chaque utilisateur peut soumettre jusqu’à deux idées par jour.
      • Un utilisateur peut commenter jusqu’à trois fois par idée.
      • Seul l’auteur d’une idée ou d’un commentaire peut le supprimer.
      • Une charte d’utilisation a été intégrée afin de formaliser les règles et assurer la conformité aux normes en vigueur.
    • Mon approche technique

      J’ai opté pour Laravel en raison de son architecture Modèle-Vue-Contrôleur (MVC) bien structurée :

      • Modèles (Models) : entités principales (idées, commentaires). Exemple : un modèle Idea décrit la structure de chaque idée et sa relation avec l’utilisateur.
      • Contrôleurs (Controllers) : gèrent la logique métier (création, modification, suppression) et vérifient le respect des règles (ex. IdeaController).
      • Routes : elles mappent les URL vers les contrôleurs (afficher, soumettre, commenter, etc.).
      • Vues (Views) : grâce à Blade, j’ai créé des pages web dynamiques (liste des idées, page de soumission, etc.).
      Laravel a également simplifié la gestion de base de données via ses migrations, et ses middlewares m’ont permis de gérer l’authentification et les autorisations.
    • Sécurisation et filtrage des données

      Après avoir implémenté les fonctionnalités principales, j’ai testé la sécurité de l’application via Burp Suite et OWASP ZAP sur Kali Linux, afin d’identifier des failles potentielles. Dans un exercice fictif, j’ai également développé un programme Java (sous Eclipse) pour traiter des données compromises exportées en CSV (application concurrente) :
      • Analyse des fichiers CSV pour détecter des motifs de code malveillant (expressions régulières).
      • Filtrage des données valides (suppression des entrées corrompues).
      • Génération d’un fichier CSV “nettoyé”, prêt à être réimporté dans l’application.
      Cette partie m’a initié à la sécurité, la manipulation de fichiers CSV et l’automatisation du “nettoyage” de données.
    • Les acquis de ce projet

      Ce projet illustre :
      • Ma capacité à employer des outils modernes (Laravel, Composer) pour bâtir une application web complète.
      • Mon sens de la sécurité informatique (recherche de failles, filtrage des données, compatibilité avec des règles métier).
      • Mon respect d’une architecture MVC claire (séparation du Modèle, de la Vue et du Contrôleur).
      • Mon aptitude à traiter un flux de données “corrompues” (programme Java) et à automatiser le process de vérification.
