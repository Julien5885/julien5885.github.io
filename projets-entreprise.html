Projet #1 (1ère année) — Comparaison de deux fichiers CSV avec Java (Eclipse)

  • Titre du projet :
    Comparaison de deux fichiers CSV
  • Contexte :
    Réalisé lors de la 1ère année du BTS SIO. Ayant un niveau débutant : quel que soit le langage informatique. J'ai peu developpé cette année là. L'informatique restait pour moi synonyme d'intêret et de curiosité. Je la voyais avec un oeil presque inconnu, comme une science infinie de part son évolution permanente. Chaque projet est vite devenu comme un défit. Je suis reconnaissant et conscient, malgrès mes lacunes, que j'ai eu beaucoup de chance d'avoir pu réaliser l'ensemble des projets qui ont suivis. Projets alors bourgeons d'opportunités. Les projets que j'ai réalisés sont loin d'être parfaits, parfois mals structurés, surtout lorsque je replonge dans les premiers : N'est pas la un signe de progrès ? J'ai beaucoup appris et j'ai encore mille fois plus à apprendre.
  • Technologies utilisées :

    • Eclipse : IDE utilisé pour coder en Java et lancer le programme. Comme tout, il faut apprendre à maîtriser son environement et Eclipse, bien maitrisé est un très bon outil pour débuter.
  • Objectif :
    L’objectif était de nous familiariser avec les bases de la programmation en Java. Le projet consistait à comparer deux fichiers CSV pour repérer à la fois les lignes identiques de celles différentes.
  • Mon rôle :
    J’ai commencé par analyser les données contenues dans chaque fichier en les comparant visuellement, puis j’ai développé un programme Java capable de lire les lignes des fichiers une par une. De comparer le texte des colonnes respectives des deux fichiers en utilisans un REGEX. Permettant ainsi de distinguer ligne par ligne, colonnes par colonnes, les différences et les similarités. Le résultat du tri s'effectuant dans un premier temps dans la console puis par la suite dans deux fichiers Excel en utilisant la bibliothèque (import) :.jxl

Projet CYBER (2ème année) — Recherche de vulnérabilités (Cellule CYBER)

  • Titre du projet :
    Recherche de vulnérabilités à partir du site YESWEHACK (Cellule Cyber)
  • Contexte :

    Réalisé en 2ème année au sein d’une cellule Cyber (3 personnes), ce projet nous a permis de découvrir la plateforme YesWeHack, spécialisée dans la mise en relation d’entreprises et de chercheurs en sécurité (bug bounty).Afin de vérifier la sécurité du site de ces entreprises. Avant de nous lancer dans l'aventure nous avons étudié cinq vulnérabilités courantes : Force Brute, Injection SQL, XSS réfléchi, IDOR et Open Redirect.

    Sur la base d’un site web cible (appelé scope), j'ai automatisé l’extraction de pages et l’analyse de formulaires afin de cibler rapidement la champs d'entrée nommés formulaires Il existe différents types de formulaires comme les champs commentaires dans lesquels peuvent être inséré des injections de code malveillantes. Ou comme des formulaires upload de fichiers : Ou un fichier peut être transmit avec lui aussi un code malveillant. Ces donc en recherchant ces différents type de formulaire dont nous pourrons cibler et organiser les suivis de vérification .
  • Technologies utilisées :

    • Kali Linux : Kali Linux permet de : Rechercher des failles de sécurité dans un site ou une application Tester si un mot de passe est facile à deviner Analyser le trafic réseau (ce qui circule sur Internet) Faire des tests de piratage éthique (appelé "pentesting") Apprendre la cybersécurité.
    • Burp Suite : audit de sécurité web (analyse, interception, automations).
    • OWASP ZAP : scan des URLs pour extraire tous les liens d’un site cible.
    • Python : script d’extraction automatique du HTML listé par ZAP.
    • Java (Eclipse) : pour développer un programme de détection et de recensement des formulaires (fichiers JAR exécutables).
    • Laravel / Composer / XAMPP : intégration des résultats en base de données et affichage via une interface web (tableau récapitulatif).
  • Objectif :

    L’objectif était de repérer rapidement les failles potentielles (champs d’entrée mal sécurisés, formulaires pouvant accepter du code malveillant). En automatisant une partie du processus, nous gagnons du temps pour les tests manuels (via Burp Suite) et nous ciblons directement les pages susceptibles d’être vulnérables (commentaires, upload, login…).
  • Mon rôle :

    J’ai conçu un programme Java (sous Eclipse, exporté en JAR) capable de parcourir les fichiers HTML récupérés par OWASP ZAP et d’y détecter différents types de formulaires. Les informations (ex. type de formulaire, nom du fichier, etc.) étaient ensuite stockées dans la base de données et exploitées dans une application Laravel. Celle-ci présentait un tableau de résultats (avec filtres par date, par mots-clés, etc.) afin de gérer le suivi et la validation des tests. Chaque ligne correspond à une page potentiellement sensible ; un champ “Actions” permettait d’enregistrer ou de supprimer les résultats des tests d’attaque.
  • Description de l’application de suivi

    • Écran de connexion : accès réservé aux membres de la cellule Cyber.
    • Page d’accueil : sélection des différents types de recherche ou résultats.
    • Page d’analyse : sélection du dossier contenant le code HTML (récupéré via ZAP). Le programme Java s’exécute en tâche de fond (ex. java -jar analysis.jar).
    • Page de résultats : tableau listant les pages, le type de formulaire, un champ commentaire et l’état de vérification (non vérifiée, en cours, validée…). Un moteur de recherche par date / mots-clés aide à filtrer les entrées.
    • Actions : l’utilisateur peut renseigner ses observations (ex. “Injection test OK”), marquer la page comme vérifiée, ou supprimer une ligne si c’est une fausse alerte.
    L’idée est de centraliser toutes les informations utiles pour conduire des tests de sécurité et vérifier la robustesse de chaque champ d’entrée face aux attaques ciblées (SQL Injection, XSS, etc.).
  • Captures d’écran suggérées :

    • Écran de connexion
    • Page d’accueil
    • Page d’analyse (sélection du dossier HTML + exécution JAR)
    • Résultats détaillés : tableau avec colonne “Nom du fichier”, “Type de formulaire”, “État de vérification”, “Commentaire”, “Actions”
    • Exemple d’alerte ou de test d’injection via Burp Suite

Projets de 2ème Année

Projet #1 (2ème année) — Recherche de vulnérabilités (Cellule Cyber)

  • Titre du projet :
    Recherche de vulnérabilités dans un cadre de bug bounty (Cellule Cyber)
  • Contexte :

    Réalisé en 2ème année au sein d’une cellule Cyber (3 personnes), ce projet nous a permis de découvrir la plateforme YesWeHack, spécialisée dans la mise en relation d’entreprises et de chercheurs en sécurité (bug bounty). Nous avons étudié cinq vulnérabilités courantes : Force Brute, Injection SQL, XSS réfléchi, IDOR et Open Redirect.

    En ciblant un scope précis (site web d’une entreprise), nous avons automatisé l’extraction et l’analyse de pages pour accélérer la détection des zones sensibles (commentaires, champs de login, upload de fichiers…).
  • Étapes techniques globales

    • OWASP ZAP pour recenser toutes les URLs d’un site cible (scope).
    • Programme Python #1 pour lister les URLs extraites et les nettoyer (formatage).
    • Programme Python #2 pour récupérer le code HTML de chaque URL et stocker chaque page en local (un fichier par URL).
    • Programme Java (Eclipse, exporté en .jar) pour analyser chaque fichier HTML et détecter la présence de champs d’entrée (formulaires) potentiellement vulnérables.
    • Application web Laravel pour centraliser et visualiser les résultats (base de données, tableau récapitulatif).
    Cette chaîne d’outils nous a permis de cibler efficacement toutes les pages nécessitant un test approfondi (par exemple via Burp Suite).
  • Technologies utilisées :

    • Kali Linux : environnement de tests d’intrusion.
    • OWASP ZAP : scan des URLs pour extraire tous les liens du site.
    • Python : deux scripts pour formater la liste d’URLs, puis récupérer le code HTML de chaque page.
    • Java (Eclipse) : programme de détection/analyse du type de formulaires, exporté en .jar pour être exécuté par l’application web.
    • Laravel / Composer / XAMPP : interface web de suivi, base de données MySQL pour gérer les résultats d’analyse, filtres (date, mots‐clés, type de formulaire).
    • Burp Suite : audit manuel (injection, XSS…) sur les pages identifiées comme sensibles.
  • Objectif :

    L’objectif final est de ne rater aucune page susceptible de contenir un champ d’entrée textuel ou un dêpot de fichier. Les champs d'entrées textuels ou de dépôt de fichier sont des types de formulaires (CV, photos,etc). Ainsi avec cette approche, nous automatisons la première étape (découverte des pages + analyse de base du code html avec une recherhce par mot-clés : Regex) pour concentrer nos efforts humains sur les tests de sécurité plus poussés (ex. injection SQL, XSS réfléchi) via Burp Suite. Cela facilite la traçabilité des failles et évite de passer à coté d'un formulaire succeptible de ne pas respecter les règles de sécurité conventionnelle. (Par exemple : nous trouvons un formulaire de type Login, via Burpsuite nous allons comprendre son mécanisme et vérifier si ce champs d'authentification est bien protégé) . En effet, un bounty peux contenir de nombreuses URLs présentes, il est possible de ne pas voir une page.
  • Mon rôle :

    J’ai développé le programme Java chargé d’analyser les fichiers HTML obtenus depuis le second script Python. Ce programme identifie les différents types de formulaires, puis injecte ces informations (nom du fichier, type de champ, etc.) dans une base de données pour qu’elles soient affichées dans l’application Laravel. J’ai également contribué à la mise en place de l’interface web (login, page d’accueil, page de configuration, page d’analyse, page de résultats filtrés). Cette interface permet de suivre l’état de vérification de chaque page et d’ajouter des commentaires sur les tests réalisés.
  • Description de l’application (interface de suivi)

    • Écran de connexion : Accès réservé à la cellule Cyber.
    • Page d’accueil : Présentation des bounties en cours et rappel des dernières analyses effectuées.
    • Saisie du chemin : L’utilisateur indique le dossier contenant les fichiers HTML (récupérés par le script Python #2). Le programme Java (.jar) est alors exécuté en arrière‐plan pour analyser ces fichiers.
    • Page de résultats : Tableau listant toutes les pages, le type de formulaire détecté, un champ commentaire, l’état de vérification, etc. On peut filtrer par date, par type de formulaire, ou par mot‐clé.
    • Actions : L’utilisateur peut marquer une page “vérifiée”, ajouter un commentaire (ex. “Injection test OK”), ou supprimer une ligne si c’est un faux positif. Chaque page listée peut ensuite être testée manuellement (ex. via Burp Suite) pour confirmer la vulnérabilité ou la rejeter.
    L’idée est de centraliser toutes les informations utiles pour mener des tests de sécurité complets et ne pas oublier de pages potentiellement dangereuses.
  • Captures d’écran (dans l’ordre de navigation) :

    • Liste et formatage des URLs (Programme Python #1)
    • Récupération du code HTML de chaque URL (Programme Python #2)
    • Écran de connexion (Application Laravel)
    • Page d’accueil
    • Saisie du chemin d’analyse (dossier HTML)
    • Page de résultats : tableau filtrable (par date, type de formulaire, etc.)
    • Sélection par type de formulaire
    • Sélection par date
    • Exemple d’états de vérification
